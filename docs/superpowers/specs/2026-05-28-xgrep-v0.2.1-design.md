## 1. 动机与范围

### 1.1 动机

v0.2.0 在 15 个 bench cell 中 14 个 PASS, 唯一边界 cell 是 `sst_heavy_low_hit/hit`:

| 指标 | 值 |
|---|---|
| v0.1 baseline (median) | 1.398 s |
| v0.2 (median) | 1.472 s |
| ratio | 0.9497x |
| % change | +5.29% (criterion CI [+3.71%, +6.79%]) |

v0.2 perf report 将其判定为 PASS (四舍五入到 0.95x, 处于 noise band 边界), 但严格意义上仍有 +5% 退化。v0.2.1 把它推到 ratio ≥ 1.00x (零退化), 同时不损失任何其他 cell。

### 1.2 根因 (T25 完成时已定位)

`sst_heavy_low_hit` fixture: 200k rows × 3 sheets, sst pool 50,000 entries, hit_density = 0.001 → 50 个 `HIT-row-N` 条目命中 pattern。

对该 fixture 跑 `pattern="HIT"` 时的执行链:

1. `sst::parse_with_early_abort(threshold=100)` 扫到 50 个 hits, 不触发 early-abort
2. 完整的 ~5MB sst XML 被 regex 解析完, ~100ms 成本付清
3. `fast_path::augment` 生成 `(?:HIT)|<v>0</v>|<v>1</v>|...|<v>49</v>`
4. 50 个 sst 索引在 3 个 sheet xml 都被引用 (sst pool 50k 循环用于 600k cell)
5. `decide()` 对每个 sheet 都返回 true → 所有 sheets 走完整 calamine parse
6. 净结果: sst::parse 的 ~100ms 是纯浪费, baseline 1.4s 上加 5%

### 1.3 范围 (硬边界)

**目标:**
- `sst_heavy_low_hit/hit` cell ratio ≥ 1.00x (硬指标)
- 其他 14 cells ratio ≥ 0.97x (不退化, 留 noise buffer; 比 v0.2 的 0.95x gate 收紧 2pp)
- geomean ratio ≥ 3.18x (不低于 v0.2.0)

**手段:**
- 把 `src/reader/sst.rs` 中 regex-based 的 sst XML 解析改成手写 byte scanner
- 顺手把 `src/reader/comments.rs` 也迁到同一个 scanner (代码一致性, 避免两份 XML 解析风格漂移)
- 新增独立 micro-bench 让 sst::parse 阶段可观测、可单独回归测试

**不在范围:**
- `fast_path::THRESHOLD = 100` 不动 — sst::parse 一变快, 它就不再是 hot spot
- CLI / output 行为不变
- `oracle.rs`, `cells.rs`, `hidden.rs`, `zip_index.rs` 不动
- 不优化内存 (允许 ±5% 浮动)
- 不引入新依赖

### 1.4 为什么不走 "降低 THRESHOLD" 的捷径

降低 `THRESHOLD` 至 ~30 让 50 hits 触发 abort, 在数学上无法把 ratio 推到 1.00x:

- abort 之前仍需扫 sst 找到第 30 个 hit
- 在 hits 均匀分布的最坏情况下, 需扫 ~60% 的 sst (~60ms) 才到 30 hits
- 节省上限 = 100ms − 60ms = 40ms, ratio 从 0.9497x → ~0.972x, 达不到 1.00x

只有让 sst::parse 本身从 ~100ms 降到 ~10ms 量级才能真正消除退化。byte scanner 是唯一稳定达标的方案。

## 2. 设计

### 2.1 新模块: `src/reader/xml_scan.rs`

定位: crate 内部的小型手写 XML scanner, 替换 sst + comments 的 regex 解析。**不公开** — `pub(crate)` 或 module-private 即可。

```rust
use std::ops::ControlFlow;

/// 遍历 `xml` 中每个 `<TAG[attrs]>...</TAG>` 元素。对每次匹配, 调用
/// `visit(attrs, body)` — `attrs` 是 `<TAG` 与 `>` 之间的字节切片,
/// `body` 是 `>` 与 `</TAG>` 之间的字节切片。
///
/// `visit` 返回 ControlFlow::Continue(()) 继续扫描,
/// ControlFlow::Break(()) 立即停止。返回 true 当且仅当 visit 主动 Break。
///
/// 约束 (与被替换的 regex 行为一致):
/// - 不匹配自闭合 `<TAG/>`
/// - 同名嵌套取第一个 close tag (与 regex `[\s\S]*?` 一致)
/// - body 原样返回, caller 自行决定是否 xml_unescape
pub fn for_each_tag(
    xml: &[u8],
    tag: &str,
    visit: impl FnMut(&[u8], &[u8]) -> ControlFlow<()>,
) -> bool;

/// 从 attrs 切片中提取 `name="..."` 的值。None 表示属性缺失。
/// 仅识别双引号形式 (与现有 regex 行为一致)。
pub fn attr<'a>(attrs: &'a [u8], name: &str) -> Option<&'a [u8]>;

/// XML 实体解转义。识别 &amp; &lt; &gt; &quot; &apos;, 其他实体保持原样
/// (与现有 `comments::xml_unescape` 行为一致)。
pub fn xml_unescape(bytes: &[u8]) -> String;
```

**实现思路:**
- `for_each_tag`: 一个 byte-level 状态机, 用 `memchr` 找 `<` 加速。匹配开标签时验证 `<tag` 后跟分界字符 (空白 / `>`), 防止 `<sst>` 误配到 `<si>` 之类的前缀冲突。
- `attr`: 用 `memmem` 找 `name="`, 然后取到下一个 `"`。
- `xml_unescape`: 顺序扫描, 命中 `&` 时尝试匹配 5 个已知实体, 未匹配则把 `&` 原样输出。

**性能预期:** 5MB sst → 5–10ms (memchr 助攻下接近 memory bandwidth, 远低于 regex 的 ~100ms)。

### 2.2 sst.rs 改写

两个函数都委托 `for_each_tag`:

```rust
pub fn parse(index: &mut ZipIndex) -> Result<Vec<String>, SearchError> {
    let Some(xml) = index.read_to_string("xl/sharedStrings.xml")? else {
        return Ok(Vec::new());
    };
    let mut out = Vec::new();
    xml_scan::for_each_tag(xml.as_bytes(), "si", |_attrs, body| {
        let mut text = String::new();
        xml_scan::for_each_tag(body, "t", |_attrs, t_body| {
            text.push_str(&xml_scan::xml_unescape(t_body));
            ControlFlow::Continue(())
        });
        out.push(text);
        ControlFlow::Continue(())
    });
    Ok(out)
}
```

`parse_xml_with_early_abort` 同构, 仅在 `hit_indices.len() > abort_threshold` 时返回 `ControlFlow::Break(())`。

**净影响:** 两个函数从各 ~20 行(含两份 regex 复制)收缩到各 ~15 行, 共享一份扫描实现。

### 2.3 comments.rs 改写

三处 regex 替换:

| 原 regex | 替换 |
|---|---|
| `re_target = Target="([^"]*comments[^"]+\.xml)"` | `for_each_tag(rels, "Relationship", ...)` + `attr(attrs, "Target")` + 在 caller 做 `target.contains("comments")` 过滤 |
| `re_comment = <comment[^>]*ref="([^"]+)"[^>]*>([\s\S]*?)</comment>` | `for_each_tag(xml, "comment", ...)` + `attr(attrs, "ref")` |
| `re_t = <t[^>]*>([\s\S]*?)</t>` | `for_each_tag(body, "t", ...)` |

`xml_unescape` 从 `comments.rs` 移除 (或保留为 thin wrapper 转发到 xml_scan), 让 xml_scan 的 byte 版作为权威实现。

### 2.4 不动的部分

- `src/reader/fast_path.rs` 一行不改, `THRESHOLD = 100` 维持
- `src/reader/mod.rs` 一行不改
- `src/reader/zip_index.rs`, `cells.rs`, `hidden.rs`, `oracle.rs` 不碰
- 公共 API 不变 — `sst::parse` 和 `sst::parse_with_early_abort` 签名一致, `comments::extract` 签名一致
- 行为字节级兼容 v0.2.0 (输出顺序、字符串内容、unescape 结果一致)

## 3. 测试

### 3.1 新增: `src/reader/xml_scan.rs` 内单元测试

约 10 个 case, 覆盖以下边界:

**`for_each_tag` 基础:**
- `<a>b</a>` → 1 个匹配, body = `b`
- `<a x="1">b</a>` → attrs = `x="1"`, body = `b`
- `<a><b>1</b><b>2</b></a>` → for_each_tag(body, "b") yields 2 个匹配
- 空输入 → 0 个匹配, 不 panic
- `<a>b` (缺少 close tag) → 0 个匹配, 不 panic
- `<abc>x</abc>` 用 `tag="a"` → 0 个匹配 (前缀冲突防护)

**`for_each_tag` 控制流:**
- visit 返回 Break 后立即停止, 函数返回 true
- visit 全程 Continue, 函数返回 false

**`attr`:**
- `x="1" y="2"` → attr(_, "y") = Some(b"2"), attr(_, "z") = None
- `x="1"` 前后有空白 → 正常识别
- `attr` 名是另一个 attr 的子串 (`name="a" name_full="b"`) → 不误配

**`xml_unescape`:**
- 5 个标准实体: `&amp;` `&lt;` `&gt;` `&quot;` `&apos;` 全部正确转换
- 未知实体 `&xyz;` 保持原样
- 无实体的 ASCII / UTF-8 多字节 (中文) 正常透传

### 3.2 保持: 现有测试

- `src/reader/sst.rs` 内 7 个单元测试 (build_hit_set + parse_xml_with_early_abort) **必须全部通过** — 它们是新实现的回归网。
- `tests/it/sst.rs` 集成测试 (对真实 xlsx 跑 parse) **必须全部通过** — 行为字节级兼容的硬门槛。
- `tests/it/` 下所有 v0.1 + v0.2 集成测试套 **必须全部通过**。

### 3.3 新增: comments 边界覆盖 (如缺)

实施阶段先 audit `tests/it/` 是否已有 comments 集成测试。如已覆盖, 不加; 如未覆盖, 加 1-2 个 fixture-based 测试 (一个普通 comment, 一个含 XML 实体的 comment)。

## 4. Bench / Gate

### 4.1 新增 micro-bench: `benches/sst_parse.rs`

criterion harness, 4 个 input case:

| case | sst entries | hit density | 期望 median |
|---|---|---|---|
| small_sparse | 5,000 | 0.001 | ≤ 3 ms |
| heavy_sparse | 50,000 | 0.001 | **≤ 30 ms** ← micro-gate |
| heavy_dense | 50,000 | 0.5 | ≤ 5 ms (early-abort 触发) |
| huge_sparse | 100,000 | 0.001 | ≤ 60 ms |

input 通过 `xtask` 现有的 fixture 生成器复用, 或直接在 bench setup 里生成合成 sst XML 字符串 (后者更简单, 不占 disk)。

**micro-gate**: `heavy_sparse` ≤ 30 ms (10x 于现 ~100ms regex 实现)。

### 4.2 端到端 bench gate (回归 v0.2 perf report 矩阵)

跑 `benches/search.rs` 全 15 cell, 对比同二进制 `XGREP_DISABLE_FAST_PATH=1` (v0.1-equiv baseline) 与默认 (v0.2.1 candidate):

| gate | 阈值 | 来源 |
|---|---|---|
| `sst_heavy_low_hit/hit` ratio | **≥ 1.00x** | 本次目标 (硬指标) |
| 其他 14 cells ratio | ≥ 0.97x | 比 v0.2 收紧 2pp |
| geomean ratio | ≥ 3.18x | 不低于 v0.2.0 |

任一 gate 未达 → 不发布, 回到设计阶段。

### 4.3 内存 smoke

```
cargo xtask measure-memory --fixture sst_heavy_low_hit
```

物理内存 delta 与 v0.2.0 一致 (±5%), 匹配数 600。

byte scanner 的内存 footprint 应当与 regex 实现接近或更低 (一个 String per `<si>` 仍需分配, 但 regex captures 的临时分配消失)。

## 5. 风险与缓解

### R1: byte scanner 正确性 bug → silent miss

最严重: byte scanner 漏读某个 `<si>` / `<t>` → sst 缺条目 → hit_set 缺索引 → fast_path 错误跳过 sheet → 用户看到的是"v0.2.0 有 600 个 match, v0.2.1 只有 599 个"。

**缓解:**
1. `tests/it/sst.rs` 对真实 xlsx 跑 `sst::parse` 并断言内容, 是新实现的硬回归网
2. 新增 xml_scan 单元测试覆盖各边界 (见 §3.1)
3. 端到端 bench 跑 match count 校验 (memory smoke 也跑, 自动比对 600 match)
4. v0.2 已有的 debug-build `oracle::assert_skipped_safely` 仍生效 — 任何"假跳过"在 debug 测试时立即 panic

无需引入额外 debug oracle (overkill, 三层防护已足够)。

### R2: byte scanner 某个形状下反而比 regex 慢

理论上有可能 — 例如 sst 含大量 `<si>` 但每个 body 都很短, 启动开销占比上升。

**缓解:** §4.1 的 4-case micro-bench 覆盖了 small/heavy/dense/huge 四种形态。任一形态退化, 在实施阶段就被 micro-bench 捕获, 不会传到 end-to-end gate 才发现。

### R3: comments 解析改动引入 regression

comments 整体覆盖率低于 sst。

**缓解:** §3.3 在实施阶段先 audit, 不足时补集成测试。comments 的输出格式 (sheet, cell, text) 比 sst 简单, 出 bug 风险也低。

### R4: 比 v0.2 收紧到 0.97x 的 gate 让其他 cell 失败

v0.2 perf report 中 14 个 cell 当前 ratio 都 ≥ 0.992x, 仅 `many_small/hit` 在 0.992x 接近 0.97 边界。byte scanner 是 sst::parse 的纯加速, 不应对 `many_small/hit` 这类非 sst-heavy cell 产生负面影响。

**缓解:** 若 `many_small/hit` 因噪声波动落到 < 0.97x, 视为 noise 而非 regression, 允许 spec 阶段把该 cell 的 gate 单独放宽到 0.95x。这点在实施 task 阶段如有需要再调整, 不阻塞设计。
