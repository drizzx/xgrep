## 1. 动机与范围

### 1.1 动机

v0.2.1 shipped 时 `sst_heavy_low_hit/hit` cell 落在 **0.970x** ratio, 比 v0.2.0 的 0.950x 推进了 +2pp, 但仍未达到 v0.2.1 spec 的原始目标 ≥ 1.00x。

v0.2.1 perf report 的架构分析定位了原因: candidate 路径的额外成本是 ~40 ms (主要是 sst::parse 阶段 + Vec<String> 分配), 而 baseline (no fast-path) 不付这个成本。即使 byte scanner 把 sst::parse 从 ~100ms 压到 ~15ms, 剩余的 ~40ms 仍然让 ratio 卡在 ~0.97x。

v0.2.2 通过两个独立但互补的优化, 把这 ~40ms 推到 ~5-10ms, 让 cell 跨过 1.00x:

- **A. 非分配 sst scanner**: 消除 `Vec<String>` 分配和 per-entry `String` 分配
- **B. workbook-shape pre-check**: 当 workbook shape 暗示 dense_bypass 必将触发时, 跳过 sst::parse 阶段本身

### 1.2 范围 (硬边界)

**目标:**
- `sst_heavy_low_hit/hit` ratio ≥ **1.00x** (硬指标 — 本次明确攻关)
- 其他 14 cells ratio ≥ 0.97x (沿用 v0.2.1 gate)
- geomean ratio ≥ 3.78x (不低于 v0.2.1)
- micro-bench `heavy_sparse` ≤ **20 ms** (比 v0.2.1 的 30 ms 收紧, 反映非分配优化)

**手段:**
- 重写 `sst::parse_with_early_abort` 用复用 String buffer + 跳过非必要 xml_unescape
- 改 `parse_with_early_abort` 返回类型: `(Vec<String>, HitSet, bool)` → `(usize, HitSet, bool)`
- 新增 `fast_path::should_skip_sst_parse(compressed_sst_bytes, sheet_count)` 启发式
- 在 `mod.rs::read_cells` 中, 当启发式触发时跳过 sst::parse, 直接 fall through 到 calamine 全 parse

**不在范围:**
- CLI / output 行为不变
- `xml_scan` 模块 API 不变
- `HitSet` 公共 API 不变
- v0.2.1 引入的 `dense_bypass` 启发式保留 (作为 pre-check 漏判后的降级路径)
- v0.2.1 perf report follow-up #3 (调优 BYPASS_* 常量) — 推迟到有真实 corpus 之后
- 不优化内存 (允许 ±5% 浮动, 沿用 v0.2.1)
- 不引入新依赖

### 1.3 为什么这两个组合能跨 1.00x

| 阶段 | v0.2.1 cost | v0.2.2 预估 |
|---|---|---|
| ZipIndex::open | ~5 ms | ~5 ms (不变) |
| sst::parse byte scan | ~15 ms | ~5 ms (跳过 xml_unescape 当 no `&` + buffer reuse) |
| Vec<String> + String alloc | ~10-15 ms | **0 ms** (消除) |
| Pre-check decision | — | ~0 ms (compressed size + sheet count 已在手) |
| **当 pre-check 触发**: 完全跳过 sst::parse | — | **~0 ms (新路径)** |
| 总 candidate overhead | ~40 ms | ~5-10 ms |

对 `sst_heavy_low_hit/hit` (baseline ~1.33s):
- v0.2.1: 1.33 + 0.04 = 1.37s → ratio 0.97x
- v0.2.2 A only: 1.33 + 0.01 = 1.34s → ratio ~0.99x
- v0.2.2 A + B (pre-check 触发): 1.33 + 0.005 = 1.335s → ratio ~1.00x+

## 2. 设计

### 2.1 A. 非分配 sst scanner

#### 2.1.1 API 变更

`sst::parse_with_early_abort` 签名:

```rust
// Before (v0.2.1):
pub fn parse_with_early_abort(
    index: &mut ZipIndex,
    pattern: Option<&Pattern>,
    abort_threshold: usize,
) -> Result<(Vec<String>, HitSet, bool), SearchError>;

// After (v0.2.2):
pub fn parse_with_early_abort(
    index: &mut ZipIndex,
    pattern: Option<&Pattern>,
    abort_threshold: usize,
) -> Result<(usize /* sst_size */, HitSet, bool), SearchError>;
```

`parse_xml_with_early_abort` (内部 helper) 同步改返回类型。

`parse` (legacy, 还在 `tests/it/sst.rs` 用) 保持不变 — 它返回 `Vec<String>` 是有用的 (测试需要 string 内容)。

`HitSet` 不变 — `len()` 仍然返回 sst 容量, `count()` 返回 hit 数。

#### 2.1.2 实现思路

```rust
fn parse_xml_with_early_abort(
    xml: &str,
    pattern: Option<&Pattern>,
    abort_threshold: usize,
) -> (usize, HitSet, bool) {
    use std::ops::ControlFlow;
    let mut sst_size = 0usize;
    let mut hit_indices: Vec<usize> = Vec::new();
    let mut text_buf = String::new();  // reused across <si> entries

    let aborted = xml_scan::for_each_tag(xml.as_bytes(), "si", |_attrs, body| {
        let idx = sst_size;
        sst_size += 1;
        text_buf.clear();  // reset to empty, KEEP capacity

        xml_scan::for_each_tag(body, "t", |_t_attrs, t_body| {
            append_t_body(&mut text_buf, t_body);  // see 2.1.3
            ControlFlow::Continue(())
        });

        if let Some(p) = pattern {
            if p.is_match(&text_buf) {
                hit_indices.push(idx);
                if hit_indices.len() > abort_threshold {
                    return ControlFlow::Break(());
                }
            }
        }
        ControlFlow::Continue(())
    });

    let mut hs = HitSet::new(sst_size);
    for &i in &hit_indices {
        hs.insert(i);
    }
    (sst_size, hs, aborted)
}
```

关键点:
- `text_buf` 在循环开始**前**分配一次, 在每个 `<si>` 开始时 `clear()` (保留 capacity, 不释放内存)
- 这把 50,000 次 String 分配压成 1 次
- 不再 `sst.push(text)` — 完全消除 Vec<String>
- pattern.is_match 在 `&text_buf` 上调用 (借用), 不需要 clone

#### 2.1.3 `append_t_body` helper

替代当前的 `text.push_str(&xml_unescape(t_body))`:

```rust
fn append_t_body(out: &mut String, t_body: &[u8]) {
    // Fast path: if t_body contains no '&', it's plain UTF-8 — push directly.
    if !t_body.contains(&b'&') {
        if let Ok(s) = std::str::from_utf8(t_body) {
            out.push_str(s);
            return;
        }
    }
    // Slow path: has entities or invalid UTF-8 — go through xml_unescape.
    out.push_str(&xml_scan::xml_unescape(t_body));
}
```

`memchr` 找 `&` 是 ~1 GB/s 量级, 对 ~10 字节的典型 `<t>` body 几乎零开销。命中 fast path 时, 完全跳过 xml_unescape 的 Vec<u8> 分配。

放在 `sst.rs` 模块私有 (不污染 xml_scan 公共 API)。

#### 2.1.4 测试调整

`src/reader/sst.rs` 现有 9 个测试 (5 个 HitSet/build_hit_set + 4 个 parse_xml_with_early_abort):
- HitSet 测试不动
- build_hit_set 测试不动 (用 `Vec<String>` 输入, 与 `parse_with_early_abort` 解耦)
- 4 个 `parse_xml_with_early_abort_*` 测试: 把 `assert_eq!(sst.len(), N)` 改成 `assert_eq!(sst_size, N)`

`tests/it/sst.rs` 不动 — 它用的是 `parse` 不是 `parse_with_early_abort`。

新增 sst.rs 单元测试:
- `parse_xml_with_early_abort_reuses_text_buf` — 跑两遍 (一遍长字符串, 一遍短字符串), 确保结果正确 (没有 stale buffer 内容泄漏)
- `parse_xml_with_early_abort_handles_xml_entities` — body 含 `&amp;`, 走 slow path, 结果正确

### 2.2 B. workbook-shape pre-check

#### 2.2.1 设计

新启发式 `fast_path::should_skip_sst_parse`:

```rust
/// Constants for the workbook-shape pre-check. When the workbook shape
/// indicates that dense_bypass will almost certainly trigger (large sst +
/// few sheets), skip sst::parse entirely — the bypass decision can be
/// inferred from the workbook shape without running the pattern.
///
/// Note: this is a SPECULATIVE shortcut. If the heuristic mis-fires on a
/// workbook with rare matches, we lose the fast-path opportunity to skip
/// non-matching sheets. The threshold values trade off (cost saved on
/// typical large-sst workbooks) vs (cost paid on outlier sparse-pattern
/// large-sst workbooks).
pub const PRESKIP_SST_COMPRESSED_BYTES: usize = 100_000;
pub const PRESKIP_SST_MAX_SHEETS: usize = 5;

pub fn should_skip_sst_parse(
    compressed_sst_bytes: usize,
    sheet_count: usize,
) -> bool {
    compressed_sst_bytes > PRESKIP_SST_COMPRESSED_BYTES
        && sheet_count <= PRESKIP_SST_MAX_SHEETS
}
```

#### 2.2.2 触发条件来源 (零成本)

- `compressed_sst_bytes`: 从 `ZipIndex` 的 zip metadata 拿 — 该 metadata 在 `ZipIndex::open` 时已扫过, 无 I/O。
- `sheet_count`: 从 `calamine::Workbook::sheets_metadata().len()` — 已经在调用 (用于 sheet 遍历)。

#### 2.2.3 在 `mod.rs::read_cells` 的接入

```rust
pub fn read_cells<'a>(
    path: &Path,
    opts: &ReaderOptions<'a>,
) -> Result<Vec<CellRecord>, SearchError> {
    let mut zip_index = ZipIndex::open(path)?;
    let want_fast_path = !opts.disable_fast_path && opts.pattern.is_some();

    // Open calamine workbook early to get sheet_count for pre-check.
    let mut workbook: Xlsx<_> = open_workbook(path).map_err(map_calamine_err)?;
    let sheets = workbook.sheets_metadata().to_vec();

    // Workbook-shape pre-check: when shape indicates dense_bypass will
    // almost certainly trigger, skip sst::parse entirely and route
    // through calamine for all sheets.
    let compressed_sst_size = zip_index
        .compressed_size_of("xl/sharedStrings.xml")
        .unwrap_or(0);
    let preskip_sst = want_fast_path
        && fast_path::should_skip_sst_parse(compressed_sst_size, sheets.len());

    let (_sst_size, hit_set, aborted) = if want_fast_path && !preskip_sst {
        sst::parse_with_early_abort(&mut zip_index, opts.pattern, fast_path::THRESHOLD)?
    } else {
        (0, sst::HitSet::new(0), false)
    };

    let dense_bypass = want_fast_path
        && !aborted
        && !preskip_sst  // already in bypass state
        && fast_path::should_dense_bypass(hit_set.count(), hit_set.len());

    let augmented = if want_fast_path && !aborted && !preskip_sst && !dense_bypass {
        Some(fast_path::augment(opts.pattern.unwrap(), &hit_set))
    } else {
        None
    };

    // ... rest unchanged
}
```

注意:
- `ZipIndex` 需要新方法 `compressed_size_of(name: &str) -> Option<u64>` (或类似), 暴露 zip metadata 中的 entry compressed size。这是个小的 ZipIndex API 加项。
- `open_workbook` 顺序略调整 — v0.2.1 是先 `sst::parse`, 再 `open_workbook`; v0.2.2 必须 swap 顺序, 因为 pre-check 需要 sheet_count。这个 swap 本身是 transparent (calamine 不读 sheet 内容直到 worksheet_range)。

#### 2.2.4 测试

新增 `fast_path.rs` 单元测试 (~6 个):
- `should_skip_sst_parse_large_sst_few_sheets` — 大压缩 sst + 3 sheets → true
- `should_skip_sst_parse_large_sst_many_sheets` — 大压缩 sst + 10 sheets → false
- `should_skip_sst_parse_small_sst_few_sheets` — 小压缩 sst + 3 sheets → false
- `should_skip_sst_parse_at_compressed_boundary` — exactly 100_000 bytes (boundary 是 strict >) → false; 100_001 → true
- `should_skip_sst_parse_at_sheets_boundary` — 5 sheets → true (boundary 是 strict <=); 6 → false
- `should_skip_sst_parse_zero_sst` — 0 bytes (没有 sst) → false

新增集成测试 `tests/it/perf_oracle.rs` (现有文件) 或 `tests/it/fast_path_specific.rs`:
- `preskip_sst_falls_back_safely_when_pattern_is_rare` — 构造 huge SST + 3 sheets + pattern 仅匹配 1 个 sst 条目 → oracle (fast-path on vs disable_fast_path) 必须 yield 相同 match (容许 ratio 上不同, 但 match count 必须 byte-exact)

#### 2.2.5 风险点详细分析

**R-preskip-1: 当 pre-check 触发但 pattern 实际只匹配少量 sst 条目, fast-path 本可以跳过部分 sheets, 现在被 pre-check 抢先 bypass 了**

具体场景: workbook 有 100,000-entry sst, 3 sheets, pattern 仅匹配 1 个 sst 条目。
- 没有 pre-check: sst::parse 找到 1 个 hit, augment 生成 `(?:pattern)|<v>N</v>`, per-sheet scan 在没引用 `<v>N</v>` 的 sheet 上返回 false → 跳过那些 sheet。
- 有 pre-check: 直接走 calamine 全 parse, 损失了 fast-path 的 sheet skip 机会。

损失估算: 假设 3 sheets 里 2 个没有 hit, 1 个有 → calamine 多 parse 了 2 个 sheets, 假设每个 ~500ms → 损失 ~1 秒。

发生概率: 实际工作簿里 "huge sst + few sheets + rare pattern" 多见于"包含很多模板/配置字符串但实际数据稀疏"的场景。不主流但存在。

缓解: 集成测试 `preskip_sst_falls_back_safely_when_pattern_is_rare` 验证正确性 (match count 一致), 接受 perf trade-off。

**R-preskip-2: ZipIndex::compressed_size_of 返回错误的字节数 (zip metadata bug 或新 zip 格式)**

缓解: `unwrap_or(0)` 兜底 — 取不到就当 0, pre-check 返回 false, 走 v0.2.1 路径。无安全风险。

### 2.3 与 v0.2.1 dense_bypass 的关系

- v0.2.1 的 `dense_bypass(hit_count, sst_len)` 保留, 是 **fallback 路径**: 当 pre-check 没触发但 sst::parse 后发现 hit set 密集时, 仍走 bypass。
- v0.2.2 的 `should_skip_sst_parse` 是 **shortcut 路径**: 在 sst::parse 之前预判, 触发时直接跳过 sst::parse。
- 触发优先级: `aborted` > `preskip_sst` > `dense_bypass`。第一个 true 即决定 augmented = None。
- 对 `sst_heavy_low_hit/hit`: pre-check 触发 (压缩 sst 大, 3 sheets), 不再走 sst::parse, 直接 calamine。新路径。
- 对 `sst_heavy_high_hit/hit` (25,000 hits): pre-check 也会触发 (同 shape)。无论 pre-check 触不触发, calamine 都会全 parse 同样 3 个 sheet (因为 fast-path 在高 hit 下也救不了)。结果等价。
- 对 `many_small/hit`: 每个文件压缩 sst 很小 (远低于 100,000 bytes), pre-check 不触发。继续走 v0.2.1 的 dense_bypass 路径。

## 5. 风险汇总

| 风险 | 严重度 | 缓解 |
|---|---|---|
| R1: text_buf 复用引入正确性 bug (借用 / 跨闭包) | High | `text_buf.clear()` 在每个 `<si>` 开头无条件调用; tests/it 集成测试是回归网 |
| R2: preskip 在 sparse-pattern + large-sst workbook 上 perf 损失 | Medium | 新增 oracle 集成测试; 接受这个 trade-off; 常量可调 |
| R3: ZipIndex compressed_size_of 不可靠 | Low | `unwrap_or(0)` fallback, 退化到 v0.2.1 路径 |
| R4: micro-bench gate 收紧到 20ms 过严 | Low | 实施阶段先测, 不达标再调 |
| R5: API change (parse_with_early_abort 返回值) 破坏外部消费者 | Low | xgrep 没发到 crates.io, 内部消费者只有 mod.rs + tests/it/sst.rs (后者用 `parse` 不是 `parse_with_early_abort`) |
