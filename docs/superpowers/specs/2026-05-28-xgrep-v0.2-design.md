## 1. 动机与范围

### 1.1 动机

v0.1 把 reader 实现为「打开 → 让 calamine 全部反序列化 → 遍历每个 cell」。对单个 100MB+ 的 xlsx，这条路径 90% 的时间花在解压 + XML SAX + 字符串 intern，而绝大多数 cell 与 pattern 无关。`rg` 在文本场景已经证明：**快不是因为 CPU 强，而是因为大部分内容根本没被读**。v0.2 把这条经验搬进 xlsx。

### 1.2 范围（硬边界）

- **目标：** 在选定 fixture 集上，相对 v0.1 tag 实测加速 ≥3x（几何平均）。低于 3x 不发布 v0.2。
- **手段：**
  - share-strings 预过滤（sst pre-scan，fast-path A）
  - sheet-xml fast-path（sheet-level byte-scan skip，fast-path B）
  - bench harness（criterion + xtask fixture 生成）

### 1.3 显式非范围

- **任何 CLI 表面变更**（除一个隐藏 env 调试开关，见 §5.2）
- **任何 layer / output 行为变更**（透明优化，行为字节级兼容 v0.1）
- **内存优化**（允许 ≤5% 退步，但不主动改善，不作硬门槛）
- streaming reader（留给 v0.3）
- 新格式（.xls / .csv / .ods）
- 上下文 `-A` / `-B` / `-C`
- 加密 xlsx 解密

### 1.4 风险

fast-path 是 silent correctness 风险面 —— 如果 fast-path 误判跳过实际有命中的 sheet，用户看到的是「v0.1 找到的 v0.2 找不到」，没有 panic 也没有 stderr。

兜底：

1. **debug shadow oracle**：debug build 里 fast-path 决策必须与 fallback 全解析等价，否则 panic（§2.3）
2. **v0.1 集成测试套全部继承**，行为变化立即被捕获（§4.1 L2）
3. **端到端 oracle 测试**，跨所有 fixture × pattern 矩阵验等价（§4.3）
4. **隐藏 env `XGREP_DISABLE_FAST_PATH=1`** 作 release 时 escape hatch（§5.2）

## 2. 优化技术细节

### 2.1 fast-path A — Shared Strings 预过滤

**机理**

xlsx zip 内 `xl/sharedStrings.xml` 是所有字符串去重池。文本类 cell 通过 `<c t="s"><v>idx</v></c>` 引用 sst 索引。如果 pattern 在整个 sst 都没命中，则**所有 `t="s"` 的 cell 都不可能命中**。

**算法**

1. open zip → 抽出 `xl/sharedStrings.xml`
2. 用一个轻量 streaming 解析（regex 提取 `<t[^>]*>([\s\S]*?)</t>` + xml unescape，复用 v0.1 reader 里 `extract_comments` 已有的逻辑模板）得到 `Vec<String>`（顺序即 sst 索引）
3. 在每个 string 上跑 pattern，命中的索引收集为 `HitSet: BitSet`
4. 把 `HitSet` 作为 reader options 的一部分传给 sheet 解析

**保守边界（必须明确）**

sst 预过滤**只**能裁掉 `t="s"` 引用类型的 cell。它**不能**裁：

- inline strings（`<c t="inlineStr">`）—— 字符串字面写在 sheet xml 里，不进 sst
- 数值（`Data::Float / Int / Bool`）—— 数字按 layer.display 走格式化，命中字符串化结果而非 sst 文本（pattern `100` 可以命中 `Data::Float(100.0)` 的 display value）
- cached values（公式 cell 的缓存值）—— 在 sheet xml 的 `<v>...</v>` 里，不进 sst
- comments —— 在独立 `xl/comments*.xml`

因此 sst 预过滤**只能用来**「跳过 `t="s"` 引用」，不能用来「跳过整个 sheet」。

### 2.2 fast-path B — Sheet-XML 字节扫描

**机理**

sheet xml 是一个文本流，里头**所有可能匹配的非-sst 内容都在**：

- inline string 的 `<is><t>...</t></is>`
- cached value 的 `<v>...</v>`（公式 cell）
- 数值的 ASCII 表示（虽然存的是 `<v>100</v>`，display 也是 `100`，字节序列一致）

外加：引用 sst 的 cell 在 sheet xml 里是 `<v>idx</v>`，idx 是数字，跟 pattern 内容无关。

所以可以做一个**两阶段**判断：

1. 在整个 sheet xml 字节流上跑 regex。完全无命中 → 该 sheet 整体跳过 calamine 解析，除非该 sheet 里有 sst 命中索引（来自 fast-path A）
2. 有命中 → 走 v0.1 同款 calamine 完整路径

**这一招对数字 pattern 也成立**，因为数字的 ASCII 形式跟 display 一致。但要小心：pattern `^100$` 这种 anchor 在 sheet xml 字节流上**不能**做严格匹配，字节流里的 `100` 周围有 XML 标签。所以 sheet-xml fast-path 用的是 **unanchored regex**（去掉 `^` / `$`）。这是 **conservative**：字节流命中可能假阳性（XML 标签里有 pattern），但绝不假阴性。假阳性的代价只是退回 fallback 全解析，正确性零损失。

### 2.3 fast-path A + B 合成

```
对每个 sheet:
    sst_hit = (HitSet 非空 AND sheet 引用了 HitSet 中的任一索引)
    sheet_xml_hit = (regex 在 sheet xml 字节流上 is_match)
    if not sst_hit and not sheet_xml_hit:
        skip sheet entirely
    else:
        full parse via calamine
```

「sheet 是否引用了 HitSet 索引」与「sheet-xml 字节扫描」可以合并到一遍 regex：把 HitSet 命中的索引也加入 pattern 一起扫描：

```
augmented_pattern = "(<原 pattern>|<v>3</v>|<v>17</v>|<v>42</v>|...)"
```

如果 sst 命中索引集 K 个（K 不太大），把 `<v>{idx}</v>` 字面 一起塞 regex；sheet-xml 单遍扫描就能同时回答「这 sheet 是否需要解析」。

**边界 case：K 很大（> THRESHOLD）**

augmented regex 编译时间反而成本爆。退化为「sst 有任意命中 → 该 workbook 所有 sheet 都解析，不做 sheet skip」。THRESHOLD 经 bench 调参，初始值 100，待 fixture 落地后实测调整。

**安全约束（关键）**

user pattern 可能包含 `<v>` / `</v>` 等 XML 字面。`fast_path::augment` 实现里：

- user pattern 用 `(?:...)` 包裹隔离
- 注入部分严格使用 `<v>\d+</v>` 字面，不引入元字符
- 用 regex 编译失败做兜底（augment 失败时退化为 unaugmented 字节扫描）

### 2.4 实现策略 — debug shadow oracle

**release**：走完整 fast-path，跳过的 sheet 不解析。

**debug**（`cfg(debug_assertions)`）：

1. 跑 fast-path 决策（得到「该 sheet 是否需要解析」）
2. 无论 fast-path 决策如何，**总是**跑 fallback（完整 calamine 解析）
3. 用 fallback 结果跟 fast-path 决策核对：
   - 如果 fast-path 决策「跳过」但 fallback 找到了命中 → panic，silent correctness bug
   - 如果 fast-path 决策「不跳过」但 fallback 没命中 → 是允许的（假阳性，仅性能损失）
4. 输出 = fallback 输出（保证 debug 行为跟 release 行为一致）

shadow oracle 是 debug-only，不影响 release 性能，但任何集成测试（`cargo test`）都会经过它。这意味着 v0.1 现有的全部集成测试是 v0.2 的 correctness suite。

## 3. 架构与模块变更

### 3.1 reader 模块拆分

v0.1 `src/reader.rs` 已经接近 600 行，双 fast-path 会再叠一层。借机把它从单文件拆成模块，否则 v0.2 后续维护会变沉。

```
src/reader/
  mod.rs          # 对外 API: read_cells(path, opts) + CellRecord + ReaderOptions
  zip_index.rs    # 一次性打开 zip + 缓存 entry 列表 + 解析 workbook.xml + .rels
                  #   抽出 v0.1 reader 里散在 extract_comments / sheet_xml_paths 的逻辑
  sst.rs          # sharedStrings.xml 流式解析 → Vec<String>; HitSet 构造
  fast_path.rs    # sheet-xml 字节扫描 + augmented regex 合成 + 判定 skip
  cells.rs        # display/cached 走 calamine 的完整路径 (v0.1 中央循环挪过来)
  comments.rs     # v0.1 的 extract_comments + xml_unescape 挪过来
  hidden.rs       # v0.1 的 hidden_row_col_for_sheet 挪过来
  oracle.rs       # debug shadow oracle 比对器 (#[cfg(debug_assertions)] 才编入)
```

**理由：** v0.1 reader 把 zip 读、workbook 索引、cell 迭代、comment 提取、hidden 检测全部堆一个文件；v0.2 还要加 sst 解析、fast-path 决策、oracle。一文件无法 hold 住。拆分后：

- 每个文件 < 200 行，单一职责
- `mod.rs` 只是 orchestrator：open zip → 走 fast-path → 决定 sheets 集 → 走 cells.rs
- 测试时 `fast_path.rs` / `sst.rs` / `oracle.rs` 可独立单测

### 3.2 reader 控制流（v0.2）

```rust
pub fn read_cells(path: &Path, opts: &ReaderOptions) -> Result<Vec<CellRecord>, SearchError> {
    let zip_index = ZipIndex::open(path)?;                          // 一次性读 workbook.xml + .rels
    let pattern = opts.pattern;                                     // matcher 从 opts 传入 (新增)
    let sst = sst::parse(&zip_index)?;                              // Vec<String>
    let hit_set = sst::build_hit_set(&sst, pattern);                // BitSet
    let augmented = fast_path::augment(pattern, &hit_set);          // Option<Regex>

    let mut out = Vec::new();
    for sheet_meta in zip_index.sheets() {
        if !opts.include_hidden && sheet_meta.hidden { continue; }
        if let Some(g) = &opts.sheet_filter {
            if !g.is_match(&sheet_meta.name) { continue; }
        }

        let xml_bytes = zip_index.read_sheet(&sheet_meta)?;
        let needs_parse = fast_path::decide(&xml_bytes, &augmented, hit_set.is_empty());

        let cells = if needs_parse {
            cells::full_parse(&zip_index, &sheet_meta, opts)?
        } else {
            // debug 模式仍跑 fallback, 仅用于校验 fast-path 决策没漏命中;
            // release 模式直接 skip, zero overhead
            #[cfg(debug_assertions)]
            {
                let fallback = cells::full_parse(&zip_index, &sheet_meta, opts)?;
                oracle::assert_skipped_safely(&fallback, &sheet_meta);
            }
            Vec::new()
        };

        out.extend(cells);
    }

    if opts.layers.contains(LayerSet::COMMENT) {
        out.extend(comments::extract(&zip_index, opts)?);
    }
    Ok(out)
}
```

**关键变化**

1. **`ReaderOptions` 新增 `pattern: Option<&Pattern>`** —— fast-path 需要 pattern 来 pre-scan，不能在 sheet 内才得到 pattern。这是 reader API 的破坏性变更，但 reader 不对外发布（lib API 不稳定），唯一调用者是 `lib.rs::search_file`，同步更新即可。如果 `pattern == None`（某些测试场景），自动 fallback 到 v0.1 路径（跳过 fast-path）。
2. **`ZipIndex` 单次 open** —— v0.1 里 `extract_comments`、`hidden_row_col_for_sheet`、`read_cells` 各自重新 `zip::ZipArchive::new()`，一个 xlsx 被 open 3+ 次。v0.2 改成 open 一次，所有 reader 子模块共享 `ZipIndex` 引用。顺手的性能修，也是模块拆分的副产物。
3. **calamine 调用集中在 `cells::full_parse`** —— calamine 只在「需要完整解析」时被调用，而不是每个文件都被强制调用。

### 3.3 xtask + benches 布局

```
xgrep/
  ├─ Cargo.toml                # workspace 化
  ├─ xtask/
  │   ├─ Cargo.toml            # 独立 binary crate
  │   └─ src/
  │       └─ main.rs           # subcommands: gen-benches, list-fixtures, measure-memory
  └─ benches/
      ├─ search.rs             # criterion benches: 调用 lib API, 不走 binary
      └─ fixtures.toml         # fixture spec (人可读, xtask 消费)
```

**Workspace 化（最小改动）**

根 `Cargo.toml` 顶层增加：

```toml
[workspace]
members = [".", "xtask"]
```

主 crate（`xgrep` 自己）仍是 root，只是把 `xtask` 作为 sibling member。Cargo 支持这种 "single-crate workspace with helper" 模式。

**fixtures.toml 描述 bench fixture 矩阵**

**xtask 入口**（`cargo xtask gen-benches`）：

1. 读 `benches/fixtures.toml`
2. 输出到 `target/bench-fixtures/{name}.xlsx`
3. 已存在且 hash（仅来源该 fixture 在 toml 里的 entry，不绑 xgrep version）一致就跳过 —— 内容只由 fixtures.toml 决定
4. 用 `rust_xlsxwriter` 生成（v0.1 已经在 dev-deps 用）

**为什么不用 build.rs**：build.rs 每次 `cargo build` 都跑，生成 100MB 文件会让 normal dev loop 变痛苦。xtask 是显式入口，只在需要 bench 时调用。

### 3.4 benches/search.rs

```rust
use criterion::{criterion_group, criterion_main, Criterion, BenchmarkId};

fn bench_pattern_in_fixture(c: &mut Criterion) {
    let fixtures = ensure_fixtures();  // 若不存在则 panic 提示用户跑 `cargo xtask gen-benches`

    let mut g = c.benchmark_group("search");
    for fx in fixtures {
        for pattern in &["foo", "张三", r"\d{4}-\d{2}"] {
            g.bench_with_input(
                BenchmarkId::new(fx.name.clone(), pattern),
                &(fx.path.clone(), *pattern),
                |b, (path, pat)| b.iter(|| run_search(path, pat)),
            );
        }
    }
    g.finish();
}

criterion_group!(benches, bench_pattern_in_fixture);
criterion_main!(benches);
```

`run_search` 调 lib API（`xgrep::search_file` 或同层入口），**不**通过 binary 启动（避开 process spawn + stdout 序列化的噪声）。

## 4. 测试策略

### 4.1 三层防御

| 层 | 工具 | 角色 |
|---|---|---|
| L1 — Correctness invariant | `cfg(debug_assertions)` shadow oracle（§2.4） | 任何 fast-path 决策必须等价于 fallback。release 关掉、debug 永开 |
| L2 — Behavior regression | v0.1 集成测试套（现有 `tests/it/`）全部继承，一个不删一个不改 | 任何用户可见行为变更立即被捕获 |
| L3 — Performance gate | criterion bench + 手动 baseline 对比 | 发布前本地跑，ratio < 3x 不发版 |

L2 是 v0.2 的**主要 correctness 防线**，L1 是 silent skip bug 的**额外**保险。

### 4.2 新增单元测试（按模块对齐）

**`reader::sst::tests`**
- `parse_sst_returns_strings_in_order`
- `parse_sst_handles_xml_unescape`（`&amp;` / `&lt;` / `&#x4e2d;` numeric refs）
- `parse_sst_handles_rich_text`（一个 `<si>` 多 `<t>` 的拼接）
- `parse_sst_empty_returns_empty`

- `build_hit_set_marks_matching_indices`

### 4.3 新增集成测试

**`tests/it/perf_oracle.rs`** —— 在 v0.1 fixture 集（basic / formula / hidden / comment / rich text / merged / dates）上，用同一组 pattern 跑 `xgrep` 拿 stdout，跟一个 oracle 比对：

```rust
// oracle 实现: 直接调 calamine + matcher 暴力扫每个 cell, 不走 fast-path
fn oracle_results(path: &Path, pat: &str) -> Vec<Hit> { ... }

#[test]
fn fast_path_matches_oracle_on_all_v01_fixtures() {
    for fx in v01_fixtures() {
        for pat in test_patterns() {
            let actual = run_xgrep(&fx, pat);
            let expected = oracle_results(&fx, pat);
            assert_eq!(actual, expected, "fast-path divergence on {}", fx.display());
        }
    }
}
```

这是 L2 的硬核形式：用一个**故意慢的 oracle**做 ground truth，跨所有 fixture × 所有 pattern 检验等价。debug shadow oracle 在 reader 内部，这个是端到端的双保险。

**`tests/it/fast_path_specific.rs`** —— 针对 fast-path 边界 case 的 fixture（这些是小 fixture，直接 `rust_xlsxwriter` 在 test 里现写）：

- `inline_strings_only_xlsx`（sst 空，全靠 fast-path B）
- `sst_with_zero_matches`（fast-path A 决策跳过所有 `t="s"` sheet）
- `formula_cached_only`（sst 空 + 公式缓存值命中）
- `pattern_at_xml_boundary`（pattern 在 cell 边界，字节扫不应假阴性）
- `pattern_contains_xml_meta`（pattern = `<v>` 时，augment 不应破坏 regex）

最后一条是**真实**的安全洞：如果 user pattern 含 `<v>` / `</v>` / 等 XML 字面，augmented regex 合成时不能让 user pattern 跟我们注入的 `<v>idx</v>` 互相干扰。这条测试就是钉死这个不变量。

### 4.4 Bench harness

**`benches/search.rs`** 跑 criterion 默认参数（100 samples，3s warmup），`cargo bench` 本地跑。CI 不跑 perf-gate，但跑 `cargo bench --no-run` 验证 harness 还能编。

**Bench 矩阵**

实际跑 fixture × pattern 的笛卡尔积（5 × 3 = 15 组），geomean 跨 15 组算。下表的 "ratio" 列是按「场景」标的最低期望（pattern 取在该 fixture 上语义匹配的那一个，即 fixture 名隐含的 pattern 类别）：

| fixture | 场景特征 | 期望 v0.2 / v0.1 ratio |
|---|---|---|
| `sst_heavy_low_hit` | pattern 在 sst 命中极少（如 `nonexistent_string`） | ≥10x（fast-path A 主战场） |
| `sst_heavy_high_hit` | pattern 命中大部分 sst（如 `the` / `data`） | ≥1x（不退步） |
| `many_small` × 100 | 100 个小文件 + CJK pattern（如 `张三`） | ≥2x |
| `formula_heavy` | 公式占主 + 不命中 pattern | ≥5x（fast-path B 主战场） |
| `inline_strings_heavy` | 全 inline string + 不命中 pattern | ≥3x（fast-path B 单兵） |

**几何平均（跨全部 15 组）≥3x 才算发布达标**，每个单项 ratio ≥0.95（不退步）。`sst_heavy_high_hit` 的 ≥1x 是「不退步」约束 —— fast-path 有启动开销，在 high-hit 场景应当接近 zero overhead。

`sst_heavy_low_hit` 配 `nonexistent_string` 与 `sst_heavy_high_hit` 配 `common_word` 是同一 fixture 的两个不同 pattern 维度；bench 矩阵把它们当作两组独立 sample 处理，不互相抵消。

### 4.5 Memory smoke check

不进硬门槛（§1.3 已确认），但 bench harness 里加一个手动 spot-check：

```
cargo xtask measure-memory --fixture sst_heavy_low_hit
# 输出: peak RSS, 跟 v0.1 baseline 对比 ±5%
```

发布前手跑一次确认没退步 ≥5%。CI 不跑。

实现：Linux 读 `/proc/self/status` 的 `VmPeak`；macOS / Windows 用 `psutil`-like crate（`memory-stats` 0.1）。

### 4.6 CI 调整

`.github/workflows/ci.yml`（v0.1 已建）增加：

```yaml
- name: Compile benches
  run: cargo bench --no-run
```

不跑 bench，只确保 harness 能编。fixture 生成（`cargo xtask gen-benches`）也不在 CI 跑 —— bench fixture 生成是 dev/release 时机，CI 跑普通 `cargo test` 用的 fixture 是 v0.1 的小 fixture（rust_xlsxwriter 临时生成），跟 bench fixture 分家。

### 5.2 隐藏 escape 通道 — 一个 env var

```
XGREP_DISABLE_FAST_PATH=1   # 关掉 sst + sheet-xml fast-path, 回到 v0.1 reader 路径
```

**理由**

- shadow oracle 只能在 debug 抓住 bug，**release 不开**
- 万一发布后用户报告「v0.1 找到的 v0.2 找不到」，没有这个 env 我们没法 A/B 验证就是 fast-path 的锅
- 不在 `--help` 里出现 = 不是公开 API，不构成兼容承诺，只在 README troubleshooting 节文档化

**实现成本**：`Config` 加一字段 `disable_fast_path: bool`，`main.rs` 读 env，reader 顶层 if 走旧路径。几十行。

### 5.3 dev 命令 — `cargo xtask`

新增，不是 user binary 的一部分：

```
cargo xtask gen-benches         # 生成 / 刷新 bench fixtures
cargo xtask list-fixtures       # 列出已生成的 fixtures (path + size)
cargo xtask measure-memory      # 跑一次 fixture, 输出 peak RSS (memory smoke)
```

xtask 只在 dev / release flow 使用，README "Development" 节文档化。

## 6. 验收契约与发布流程

### 6.1 发布 checklist

每次 v0.2.x 发布前依次跑：

1. `cargo fmt --check`
2. `cargo clippy --all-targets -- -D warnings`
3. `cargo test --release`（release 模式才能验证不依赖 debug oracle 的行为）
4. `cargo test`（debug 模式，触发所有 shadow oracle，任何 fast-path 误判会 panic）
5. `cargo xtask gen-benches`（确认 fixture 全 OK）
6. `cargo bench`（本地，跑完整 bench 矩阵）
7. 把 bench 报告跟 `bench-baselines/v0.1/` 对比
8. **要求**：几何平均 ratio ≥3x 且每个单项 ratio ≥0.95
9. `cargo xtask measure-memory`（memory ±5% 不退步）
10. Commit bench 报告到 `docs/superpowers/perf/v0.2-bench-report.md`
11. 打 tag `v0.2.0` 发布

**任意一条不满足 → 不发版**，回去找原因。

### 6.2 基线快照

`bench-baselines/v0.1/` 一次性生成（v0.2 第一个任务）：

```bash
git checkout v0.1.0
cargo xtask gen-benches
cargo bench --bench search -- --save-baseline v0.1
cp -r target/criterion/.../v0.1 bench-baselines/v0.1/
git checkout main
git add bench-baselines/
git commit -m "perf: snapshot v0.1 bench baseline for v0.2 comparison"
```

之后 v0.2 任何阶段都可以：

```bash
cargo bench --bench search -- --baseline v0.1
```

criterion 自动算 ratio + 给出 confidence interval。

### 6.3 release notes 模板

```markdown
# xgrep v0.2 — Engineering release: faster reader

xgrep v0.2 is a drop-in replacement for v0.1 with internal reader optimizations.
No behavior changes. No new flags. No new dependencies for users.

## Performance

On the v0.2 bench suite (see docs/superpowers/perf/), v0.2 is N× faster than
v0.1 on geometric mean. See full table in the perf report.

## Internals (for contributors)

- Reader uses a shared-strings pre-filter to skip cell parses when the pattern
  cannot possibly match string-typed cells.
- Reader uses a sheet-XML byte scan to skip whole-sheet parses when neither
  the pattern nor any referenced shared string appears in the sheet bytes.
- Both optimizations are validated by a debug-build shadow oracle that
  asserts equivalence with the v0.1 reader on every test fixture.

Set XGREP_DISABLE_FAST_PATH=1 to force the v0.1 code path (troubleshooting).
```

### 6.4 失败模式与回退

| 情况 | 处理 |
|---|---|
| bench 几何平均 < 3x | 不发版。在 plan 里加 task 调优，重跑 |
| 任一 fixture ratio < 0.95（优化反而拖慢） | 不发版，排查为啥 |
| shadow oracle 在某 fixture 上 panic | 不发版，修 fast-path 决策逻辑 |
| 发布后用户报「v0.1 能找到 v0.2 找不到」 | 1) 让用户 `XGREP_DISABLE_FAST_PATH=1` 复测；2) 复现的 xlsx 加进 fixture；3) 修 + v0.2.1 |
| memory 退步 > 5% | spot 排查（大概率是 sst 缓存 / `ZipIndex` 持有过多 entries），调整后重测 |
